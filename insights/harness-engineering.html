Harness Engineering

The Shift From Prompting Models to Governing Execution

TX

Teresa Xu

Product Designer & AI Strategist

There's a quiet shift happening in how teams build with AI. Not in the models themselves (those keep improving regardless), but in the layer of engineering surrounding them. The question used to be: how do we prompt the model well? Now, increasingly, it's: how do we govern what the model is allowed to do?

How the Engineering Focus Has Shifted

In 2023, a good AI product was largely a good prompt. The model was a text interface and the craft was in the asking: getting it to be specific, grounded, useful. Teams invested in prompt libraries and prompt engineers. It worked, until it didn't.

As models improved, the prompt stopped being the bottleneck. What the model could reason over became the constraint: what it could retrieve, remember, and fit in its window. RAG pipelines replaced prompt documents as the thing worth optimising.

A third shift is now underway, quieter than the other two but more consequential. When a model moves from responding to acting, the real leverage point isn't what you tell it. It's what you let it do.

When Models Started Acting

With early language models, the worst thing that could happen was a confident wrong answer. You read it, you moved on.

The calculus changed when models gained tools. Now they edit files, run shell commands, call APIs, write to databases. A bad output isn't just something to correct; it's a side effect that may already be in production. The question shifts from is this answer good? to something harder: should this action have been possible at all?

That's what harness engineering addresses. The term was coined by Mitchell Hashimoto in February 2026 and formalised by OpenAI and Thoughtworks shortly after, but the underlying need predates the name. Teams had been building it ad hoc for months before anyone knew what to call it.

When Instructions Aren't Enough

Before harness engineering had a name, most teams reached for what felt like the natural solution: skills. Write a cleanup skill, a review skill, a testing skill. Register it as a capability the agent can call. Clean and composable.

The problem is subtle but important. A skill is still just a prompt: structured and reusable, but fundamentally a request. The model decides whether to invoke it, when, and whether the result met the bar. An agent that already believes its output is correct won't call the review skill. You've built a governance mechanism that only activates when the system agrees it's needed.

This is the core tension. Any instruction living inside the prompt competes with everything else in context. The model weighs it, reasons about it, and sometimes deprioritises it. That's not a failure mode; it's how language models work. It just means that prompts and skills are the wrong tool for enforcement.

before finishing."

A harness doesn't ask for compliance. It enforces regardless of what the model thinks. Skills still have a valid role (extending what a model can do is genuinely useful) but they stop being governance the moment the model has discretion over whether to invoke them. When a harness forces a skill to run unconditionally on every output, that skill has become a constraint, not a feature. And a constraint is what was needed all along.

Agent = Model + Harness. If you're not the model, you're the harness.

The Five Operational Layers

A harness isn't a single mechanism. It's a set of concerns that have to be addressed together for an agentic system to be reliably safe to operate.

1
Constraint
What can it do?

Defines the execution boundary. Without it, agent autonomy becomes uncontrolled side effects.

2
Guidance
What should it do?

Shapes how the model moves through a task: not just instructions, but defined workflow stages and output schemas.

3
Validation
Did it actually work?

Correctness verified externally. The model should not certify its own output.

4
Correction
What happens on failure?

Treats failure as expected. Validation without correction only detects problems; correction drives convergence.

retry error feedback scope narrowing escalation
5
Human Gates
What needs a human?

Reserves irreversible decisions for people. Not anti-automation: a mechanism for appropriate human authority.

data deletion schema migration prod writes infra changes

Deterministic vs. Inferential Validation

Not all validation is equal. Some correctness checks have definitive answers: pass or fail. Others require judgment, and that's where things get interesting.

Deterministic
Prefer when possible
  • ESLint / static analysis
  • TypeScript compiler
  • Unit tests
  • Schema validation
  • Forbidden-file checks

Predictable, fast, cheap. Same input always produces the same result. No model judgment involved.

Inferential (LLM-based)
Use when deterministic can't
  • AI code reviewer
  • Semantic evaluation
  • Architecture critique
  • Style consistency review

More flexible, catches unknown problems. Slower and less reliable. Still involves model judgment.

The rule of thumb is to reach for deterministic checks first. They're fast, cheap, and perfectly consistent: the same input always produces the same result. Inferential validation is more flexible and can catch things no fixed rule would anticipate, but it reintroduces model judgment into the verification step.

The Strongest Boundaries Are Invisible

There's a counterintuitive principle that becomes obvious once you've watched an agent reason around a prompt restriction: the most reliable constraints are ones the model never encounters.

A "don't do X" instruction lives inside the model's reasoning context. It competes with other instructions, can be misread, and can sometimes be talked around by a sufficiently crafted input. Move that constraint to the infrastructure layer (a sandbox permission, a policy engine, a validator that runs before output is accepted) and it becomes invisible to the model entirely.

Separate Generation from Validation

We built code review, QA, and auditing into software development for a reason: people are poor judges of their own work. Not because they're careless, but because the act of building something shapes how you perceive it. Your brain fills in the gaps, reads the intention rather than the text.

The same dynamic applies to AI systems. A model that generates output and then evaluates it is operating under the same bias; its assessment is coloured by the same reasoning that produced the output in the first place. This is why separating generation agents from review agents matters. Some teams take it further, using a different model vendor at the evaluation step to eliminate correlated blind spots entirely.

The Harness Is Already Around You

If you've used Cursor or Claude Code, you've already experienced a harness, though you may not have thought of it that way. The file scope controls, the shell access limits, the confirmation prompts before destructive actions, the error feedback loops: none of that is model behaviour. It's harness design, baked into the platform by the people who built it.

This is what explains something a lot of developers notice but struggle to articulate: the same underlying model behaves differently across tools. The weights are identical. What differs is what surrounds them. As foundation model capabilities converge (and they are converging) the harness is becoming the product.

Harnesses Won't Disappear. They'll Become Infrastructure.

It's a reasonable question: if models keep improving, won't the harness eventually become unnecessary? Won't better training just bake in the right behaviours?

Partly. Models are already being trained on the failure patterns that harnesses currently catch, and that reduces the need for explicit configuration. Behaviour that required a custom hook in 2024 is sometimes a default today.

But capability and consequence expand together. The more a model can do autonomously, the more consequential its mistakes. The governance ceiling rises in proportion to what gets automated. What changes isn't the need for a harness; it's who builds it. The logic moves from custom configuration into platform defaults, from settings files into infrastructure. Less visible, not absent. It becomes the environment.

Sufficient Control, Not Maximum Control

There's a failure mode on the other end of the spectrum. A harness that's too tight, with excessive rules, conflicting hooks, and a system prompt bloated with instructions, degrades the system it was meant to protect. Context gets consumed by governance overhead. Instructions conflict. The model becomes harder to work with, not easier.

The right mental model isn't maximum control. It's sufficient control. Start sparse. Add constraints only when you've observed real failures. Reach for deterministic validation first because it's cheap and consistent. Automate the things humans keep doing manually. And prune constraints as models improve; a rule that was necessary last year might be unnecessary overhead today.

A Familiar Pattern, a New Surface

Capability boundaries, sandboxing, validation, separation of duties: none of these ideas are new. Software engineering has relied on them for decades. What's new is the surface they're being applied to: systems that reason and act, that make judgment calls without a human in the loop, that do real work inside production environments.

The engineering patterns carry over. What takes time is recognising that they need to. The instinct with AI is still often to tweak the prompt, to improve the model, to invest in better responses. The harness gets treated as optional configuration, something to add later once the basics are working.

But the basics don't work without it. That's the shift.
