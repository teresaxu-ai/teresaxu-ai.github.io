The Golden Rules for Writing AI Skills

A practical guide to what makes a skill worth building — and when not to build one at all

TX

Teresa Xu

Product Designer & AI Strategist

If you've read the harness engineering post, you already know the key distinction: a skill is guidance, not enforcement. It lives inside the model's reasoning space, which means the model decides whether to invoke it, when, and whether the result met the bar.

This matters before anything else, because people sometimes expect skills to do everything. A skill is pure instruction — it has no way to reach outside the model's reasoning context. Understanding that boundary is the foundation for building skills that actually work.

First: Understand What a Skill Can't Do

If your workflow needs to touch something external, you're probably not looking for a skill. The three most common misroutes:

A skill can tell the model when and how to use an MCP tool or invoke a platform capability — but it can't substitute for one. The connection has to exist first. Trying to write a skill that “integrates with Salesforce” or “queries the data warehouse” will always fall short if the underlying integration isn't there.

What a Skill Is

A skill is a procedural instruction set that the model follows when it decides to invoke it. It is not a plugin, not a config file, not a workflow. It's a structured document telling the model how to approach a task — what to ask, what to read, what to generate, and in what order.

One thing to understand about invocation: the model reads all available skill descriptions at the start of a conversation and decides autonomously whether to consult one. That decision happens entirely through the description field in the frontmatter. Write a weak description and the model will skip the skill even when it's exactly what's needed.

The description field is the single most important line in your skill. It's the only thing the model reads before deciding whether to invoke the rest.

If the model doesn't trigger the skill automatically, most tools let you invoke it explicitly — in Claude Code, that's /skill-name. That's a useful escape hatch during development and for less common workflows — but if you're relying on it regularly, it's a signal the description needs work.

Rule 1: One Skill, One Job

A skill should do exactly one thing, and its trigger should be unmistakable. If you can't write the trigger condition in a single sentence, the scope is too broad.

Good: “Generate a structured client report from a data export, following the firm's standard template.”

Bad: “Help with documents, reports, data exports, and anything the client needs to review.”

Narrow scope forces specificity. Broad scope produces vague instructions the model interprets inconsistently — and creates ambiguous triggers that make the wrong skill fire.

Rule 2: The Description Field Is Your Primary Lever

The description in YAML frontmatter is the single most important line in your skill. Models tend to undertrigger — skipping skills for simple tasks they can handle directly, even when the description matches. Your description needs to be specific enough to catch cases where the skill genuinely adds value.

Be slightly pushy. Instead of “Populate a benchmark template from client files,” write: “Use this whenever the user wants to fill a benchmark template, populate structured input files, or map client data into a standard format — even if they don't explicitly name the template.” This compensates for a known undertriggering bias.

After building a skill, test the description deliberately. Generate realistic trigger and non-trigger queries — especially near-misses — and check whether the model fires when it should and holds back when it shouldn't.

Rule 3: SKILL.md Is Procedure, Supporting Files Are Data

SKILL.md should be a phased workflow — decisions, actions, branches. Not lookup tables, data schemas, template libraries, code snippets, or large reference content. That belongs in supporting files loaded on demand:

SKILL.md ← workflow logic references/field-mappings.md ← data reference, read in Phase 2 references/output-schemas.md ← format specs, read in Phase 3 assets/report-template.docx ← reusable artifact, copied into output scripts/transform.py ← helper script, run when needed

Things that don't belong inline in SKILL.md: raw CSS, JSON schemas, SQL queries, example datasets, API response shapes, long enumeration lists, boilerplate code of any kind. If it's data rather than logic, it goes in a supporting file. Aim to keep SKILL.md under 500 lines.

Rule 4: Use Phases, Not Paragraphs

A phase-based skill is reliably followable. A skill written as prose is not.

P0
Detect

Identify input type and invocation mode

P1
Gather

Ask for all required inputs in one structured block

P2
Load & Validate

Read relevant supporting files, validate inputs against schema

P3
Generate

Produce output using loaded references and validated inputs

P4
Deliver

Summarise decisions made, hand off artifact, offer next steps

Phases also serve as context-load boundaries — you only read what the current phase needs, not everything upfront. This keeps the model's working context lean and prevents early phases from being distracted by material they won't use until later.

Rule 5: Batch All User Interactions

Every round trip costs time and breaks flow. Present all required inputs in a single structured block. If your skill needs four things before it can proceed, ask for all four together.

The exception is genuine branching — if the answer to one question determines whether another is relevant, asking in stages is fine. But default to batching.

Rule 6: Label Non-Negotiables — and Explain Why

Every skill has invariants that must hold regardless of user input. Flag them explicitly (NON-NEGOTIABLE, ⚠ ALWAYS) and repeat them at the point of temptation, not just at the top.

Why the reason matters

“Always use the client's cost column, not the revenue column” is weaker than “The benchmark template expects costs in column D — using revenue instead silently produces wrong ratios that won't be caught until client review.” The second gives the model the context to defend the constraint rather than work around it.

Rule 7: Ship Reusable Artifacts, Not Just Instructions

A skill can include ready-to-use scripts, templates, and reference artifacts — not just prose instructions. A transformation script that normalises an input format, a report template with the right headers already in place, a schema file that validates output structure: all of these are more consistent than asking the model to reconstruct them from scratch each time.

The signal to watch for: if your test runs all independently produce the same helper — a normalize_columns.py, a validate_schema.json — that artifact belongs bundled in the skill.

Rule 8: Document Failure Modes Inline

Put gotcha blocks immediately adjacent to the steps that can fail:

⚠ Gotcha: If the source file has merged cells or non-standard column headers, the mapping script will silently produce wrong output. Check for these before running — ask the user to confirm column names match the expected schema.

This turns the model into a proactive guide rather than a reactive troubleshooter.

A Skill Isn't Done When You Write It

Writing the SKILL.md is a first draft. The only way to know if it works is to run it.

Come up with two or three realistic test prompts — the kind of thing an actual user would type. Run them. Watch what the model does. Look for: Did it read the right files at the right phase? Did it follow the invariants? Did it invoke the skill at all? If not, fix the skill and repeat.

One useful signal: if all your test runs independently produce the same helper script, that script belongs bundled in the skill. Read the transcripts, not just the final outputs — the process reveals what the instructions are and aren't doing.

Iteration is the job. A skill that hasn't been tested on real prompts is a hypothesis.

When Not to Write a Skill

Four situations where a skill is the wrong answer:

event-driven
Automated behaviors belong in hooks
“Always run tests before committing.” These fire in response to events. Use settings.json hooks.
global convention
Global conventions belong in project instructions
“Always use snake_case.” These apply to every interaction — put them in CLAUDE.md.
must enforce
Enforcement belongs in the harness
If it must happen regardless of what the model thinks, move it to a validator or pre/post hook.
one-time
One-time setup doesn't need a skill
A skill implies repeated use. Three paragraphs in your project instructions is the right answer.

The Case Against Too Many Skills

Once you understand skills, the temptation is to skill-ify everything. Resist it.

Too many skills creates discovery failure (ambiguous triggers, wrong skill fires), overlap and contradiction (two skills handling the same domain give inconsistent instructions), and maintenance overhead (a stale skill confidently gives outdated instructions, worse than no skill).

A good heuristic: if you could accomplish the same outcome with three paragraphs in your project instructions file, do that instead. Five well-maintained skills are worth more than twenty half-finished ones.

The Shape of a Good Skill

Narrow in scope. Phased in structure. Lazy in data loading. Honest about invariants — with reasons, not just labels. Rich in reusable artifacts. Tested against real prompts before you ship it.

And understood for what it is: a tool call the model chooses to make, not a rule the system enforces. Design accordingly.
