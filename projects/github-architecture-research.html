GitHub Architecture Research

An MCP server that goes directly to GitHub, reads how real production systems are built, and synthesises the findings into an actionable architectural brief — tradeoffs, failure modes, and MVP vs. scaling guidance included.

Role

Developer

Timeline

2025

Stack

TypeScript, Node.js, MCP, GitHub API

Category

MCP · Dev Tools · AI

The Problem It Solves

When you're starting a new project, you typically Google "how to build X" and get blog posts, tutorials, and opinionated takes. What you actually want is: what do real production systems that solve this problem actually look like?

This tool goes directly to GitHub, finds 10–20 production repositories in your problem space, reads their actual code structure and dependencies, and synthesises the findings into a structured architectural brief — not a list of repos, but a build plan that tells you what approaches exist, which one fits your situation, where systems commonly break, and what you need to decide yourself.

How It Works

The MCP server handles data (4 GitHub API tools + a structured prompt). Claude handles intelligence (pattern extraction, synthesis, recommendations). Together, they run a 6-step workflow entirely autonomously:

1

Decompose your idea into technical primitives

e.g. "google maps scraper" → web-scraping, browser-automation, data-extraction

2

Search GitHub in 3 rounds

Round 1: topic:<primary-tag> · Round 2: topic:<adjacent-tag> · Round 3: behaviour keywords

3

Filter candidates

Active within last 2 years (rolling), 10+ stars, excludes demo / tutorial / homework repos

4

Auto-select top 10–20 repos

Ranked by relevance, stars, README depth, and tech diversity

5

Deep analysis of each repo

get_repo_structure → file tree; get_file → package.json / go.mod / requirements.txt. Code beats README: manifest is ground truth.

6

Synthesise into a full architectural report

Key takeaways, recommended architecture, approaches compared, build plan staged by MVP / scaling / future

Output Structure

The report is always structured markdown, starting with the most important findings first:

SectionWhat it covers
🔑 Key TakeawaysTop cross-repo lessons — always first, before any detail
🏗️ Recommended ArchitectureOpinionated recommendation + data flow diagram
⚖️ Approaches Considered2–3 alternatives with failure modes and explicit tradeoffs
🗺️ Project Build PlanStages tagged mvp / scaling / future, each with alternatives, key decisions, and failure modes
♻️ Reusable PatternsDirectly liftable patterns with when/when-not-to-use
🚀 Where to StartConcrete first step grounded in the research
🚫 What Research Can't Tell YouSpecific gaps requiring your own domain knowledge
📦 Repos AnalysedList of sources used

MCP Tools

Four GitHub API tools do the data work. Claude does the thinking.

ToolWhat it does
search_reposSearch GitHub using topic tags and behaviour keywords. Auto-applies a rolling 2-year recency filter and 10-star minimum.
get_repo_overviewFetch README + metadata (stars, language, topics, license) for a repo
get_repo_structureList files and folders at any path within a repo
get_fileRead the raw content of any file (package.json, go.mod, requirements.txt, etc.) up to 1 MB

What I Learned Building It

MCP as a thin data layer

The cleanest design decision was keeping the MCP server purely about data retrieval and delegating all intelligence to the LLM. The four tools map directly to what you'd do manually when researching a codebase — search, read overview, explore structure, inspect files. No logic, no scoring, no heuristics in the server itself. That separation made the system easy to reason about and easy to debug.

Code over README

Early versions leaned on READMEs too heavily. The problem: README quality varies wildly. A repo with a sparse README might still be the best-engineered example in the set. Switching to package.json / go.mod / requirements.txt as the ground truth for dependencies and tech stack produced much more reliable analysis. The rule became: if the README and the manifest disagree, trust the manifest.

Structured output from the start

Requiring the final report to start with Key Takeaways — before any detail — was a forcing function for the model to synthesise rather than summarise. Without that constraint, the output tended toward a repo-by-repo list rather than cross-cutting insights. The section order in the prompt is doing real work.

Example project ideas it handles well

  • A SaaS platform for scheduling social media posts with team collaboration
  • A web scraper that extracts business leads from Google Maps and exports to CSV
  • A real-time multiplayer whiteboard app with cursor presence
  • A personal finance tracker with bank statement import and categorisation
